# Alert Webhook Engine Design Document

## Overview

The Alert Engine is a standalone system designed to handle threshold-based alerts across different components of the FlexPrice platform. The system implements a simplified three-level threshold hierarchy (Endpoint -> Entity -> Tenant) with clear precedence rules. Alerts can be triggered via API endpoints.

The system introduces dedicated Alert tables to store threshold configurations and alert states, making it more efficient to manage and track alerts across the platform. When thresholds are provided at the endpoint level, the system bypasses checking entity and tenant level thresholds, optimizing the evaluation process.

## Core Concepts

### 1. Entity Types and Metrics

Each alert in the system is associated with a specific entity type and metric combination. These are defined using the `EntityMetric` type to ensure type safety and validation.

```go
// Entity Types and Metrics are defined in types.EntityMetric or model.go
type EntityMetric string

const (
    // Wallet Entity Metrics
    WalletOngoingBalance  EntityMetric = "wallet.ongoing_balance"
    WalletCurrentBalance  EntityMetric = "wallet.current_balance"
    
    // Entitlement Entity Metrics
    EntitlementUsage      EntityMetric = "entitlement.usage"
    EntitlementLimit      EntityMetric = "entitlement.limit"
    
    // Add other entity types and metrics as needed
)
```

Each entity type can have multiple metrics that can be monitored. For example:
- Wallet entity can monitor ongoing_balance, current_balance
- Entitlement entity can monitor usage, limit
- Each new alert requirement will be added as a new entity type with its specific metrics

## Configuration Resolution

### 1. Priority Rules

1. **Endpoint Level (Highest)**
   - Set via API request with check_against_threshold
   - Overrides all other configurations
   - Used for one-time checks or scheduled evaluations

2. **Entity Level**
   - Entity-specific configuration
   - Stored in Entity Alerts Configs table
   - Used if no endpoint threshold provided

3. **Tenant Level**
   - Tenant-wide configuration
   - Stored in Tenant Alerts Configs table
   - Used if no endpoint threshold and no entity config


### 3. Alert Processing Flow

The alert processing system handles two distinct scenarios based on the presence of a threshold in the AlertCheckRequest:

#### Scenario 1: Check Against Threshold Present

When a specific threshold is provided in the request:

1. **Skip Configuration Lookup**
   - Do not fetch thresholds from TenantEnvAlertConfigs or EntityAlertConfigs tables
   - Use the provided threshold directly for comparison

2. **Entity Processing**
   For each tenant × environment combination:
   - Fetch all entities matching req.entity.type
   - For each entity:
     a. Verify alert enablement status
     b. If alert enabled:
        - Compare metric value against provided threshold
        - If threshold breach or resolution detected:
          * Validate state transition (OK ↔ IN_ALARM)
          * If transition valid, publish appropriate webhook event

#### Scenario 2: Check Against Threshold Absent

When no threshold is provided in the request:

1. **Tenant-Environment Level Processing**
   For each tenant × environment combination:
   - Fetch alert configuration (threshold, enabled status) for entity type and metric
   - If configuration missing:
     * Error: "Alert config not set for entity type & metric within env of tenant"
   - If alert disabled:
     * Error: "Alert disabled for entity type & metric within env of tenant"

2. **Entity Level Processing**
   If tenant-level alert enabled:
   - Fetch all entities matching req.entity.type
   - For each entity:
     a. Fetch entity-specific alert configuration
     b. If alert disabled for entity:
        * Skip to next entity
     c. If alert enabled:
        * Check for entity-specific threshold
        * If entity threshold exists:
          - Compare metric against entity-level threshold
        * If no entity threshold:
          - Compare metric against tenant-env level threshold
        * On threshold breach or resolution:
          - Validate state transition (OK ↔ IN_ALARM)
          - If transition valid, publish appropriate webhook event

3. **State Transitions**
   - Track transitions between OK and IN_ALARM states
   - Publish different webhook events for:
     * Alert triggering (OK → IN_ALARM)
     * Alert recovery (IN_ALARM → OK)
